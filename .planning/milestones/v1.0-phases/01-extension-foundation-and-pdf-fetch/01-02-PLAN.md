**src/offscreen/offscreen.js** — PDF fetch and storage logic:

**Message listener** (top-level):
```javascript
chrome.runtime.onMessage.addListener((message, sender, sendResponse) => {
  if (message.type === 'fetch-pdf') {
    fetchPdfWithRetry(message.pdfUrl, message.patentId);
  }
  return true;
});
```

**fetchPdfWithRetry(pdfUrl, patentId, retries = 1)** async function:
1. Try `fetch(pdfUrl)`
2. If response not ok, throw `Error(`HTTP ${response.status}: ${response.statusText}`)`
3. Get `response.blob()`
4. Call `storePdfInIndexedDB(patentId, blob)`
5. Send success message: `chrome.runtime.sendMessage({ type: 'pdf-fetch-result', success: true, patentId })`
6. On error: if retries > 0, wait 1 second then recursively call with retries - 1 (one silent retry per user decision)
7. On final failure: send `chrome.runtime.sendMessage({ type: 'pdf-fetch-result', success: false, patentId, error: error.message })`

**openDb()** function:
- Open IndexedDB database named `patent-cite-tool`, version 1
- In `onupgradeneeded`: create object store `pdfs` with keyPath `patentId`
- Return promise resolving to db

**storePdfInIndexedDB(patentId, blob)** async function:
- Open db via `openDb()`
- Write `{ patentId, pdf: blob, timestamp: Date.now() }` to `pdfs` store
- Return promise resolving on `tx.oncomplete`

**Important notes:**
- Offscreen documents only have access to `chrome.runtime` among extension APIs. No chrome.storage, no chrome.tabs. IndexedDB is a web API and works fine.
- Use message type strings directly (no import — offscreen document cannot use ES module imports from extension files reliably). Define them as local constants: `const FETCH_PDF = 'fetch-pdf'` and `const PDF_FETCH_RESULT = 'pdf-fetch-result'`.

- Store the promise in `creatingOffscreen`, await it, then set `creatingOffscreen = null`

**Update the MSG.PDF_LINK_FOUND handler:**
The existing handler from Plan 01 stores patent info with status FETCHING. After that storage write, add:
1. Call `await ensureOffscreenDocument()`
2. Send message to offscreen: `chrome.runtime.sendMessage({ type: 'fetch-pdf', pdfUrl: message.pdfUrl, patentId: message.patentId })`

**Add MSG.PDF_FETCH_RESULT handler** to the existing onMessage listener:
- Read `currentPatent` from `chrome.storage.local`
- If `message.success`:
  - Set `patent.status = 'ready'`
  - Clear badge: `chrome.action.setBadgeText({ text: '' })`
  - Save back to storage
- If `!message.success`:
  - Set `patent.status = 'error'`, `patent.error = message.error`
  - Set badge: `chrome.action.setBadgeText({ text: '!' })`, `chrome.action.setBadgeBackgroundColor({ color: '#EF4444' })` (red)
  - Save back to storage

Per user decisions:
- Badge error indicator (red) on fetch failure
- "PDF unavailable" (amber badge, already from Plan 01) is distinct from "fetch error" (red badge)

**src/popup/popup.js** — Read status and render:
On DOMContentLoaded:
1. `chrome.storage.local.get('currentPatent')` to get patent state
2. If no currentPatent data: show "Navigate to a US patent on Google Patents to begin." with class `status-idle`
3. If status is `ready`: show "PDF ready for [patentId]" with class `status-ready`
4. If status is `fetching`: show "Fetching PDF for [patentId]..." with class `status-fetching`
5. If status is `error`: show "Error fetching PDF for [patentId]" with error detail below, class `status-error`. Per user decision: user clicks icon to see brief error message.
6. If status is `unavailable`: show "No PDF available for [patentId]" with explanation "This patent does not have a downloadable PDF link." class `status-unavailable`. Per user decision: "PDF unavailable" is a distinct state — user understands it's a data limitation, not a bug.

Display the patent ID with class `patent-id` for monospace emphasis.
