## Phase Details

### Phase 5: Silent Mode
**Goal**: Users can copy patent text with citation appended using a keyboard shortcut, with no popup required
**Depends on**: Phase 4 (v1.0 complete)
**Requirements**: SLNT-01, SLNT-02, SLNT-03, SLNT-04, SLNT-05
**Success Criteria** (what must be TRUE):
  1. User can select "Silent" as a trigger mode in the extension popup settings
  2. When silent mode is active and user presses Ctrl+C on highlighted patent text, the clipboard contains the selected text with a column:line citation appended (e.g., `"text. 4:12-15"`)
  3. When the patent prefix setting is enabled, the silent citation includes the prefix (e.g., `"text. '123 Pat., 4:12-15"`)
  4. When silent mode is active and a published application paragraph is highlighted, Ctrl+C appends the paragraph citation
  5. When match confidence is low or no match is found, clipboard contains plain text only and a toast notification explains why the citation was not appended
**Plans**: 2 plans

Plans:
- [x] 05-01-PLAN.md — Settings + core clipboard interception (silent trigger mode, copy event handler, pre-computation)
- [x] 05-02-PLAN.md — Toast notification UI (success pill, failure pill, wiring into copy handler)

### Phase 6: USPTO API Fallback
**Goal**: Patents that Google Patents cannot serve are resolved by fetching the eGrant PDF via a Cloudflare Worker that orchestrates the USPTO Open Data Portal API
**Depends on**: Phase 4 (v1.0 complete) — independent of Phase 5
**Requirements**: UPTO-01, UPTO-02, UPTO-03
**Success Criteria** (what must be TRUE):
  1. Cloudflare Worker is deployed and reachable from a real extension context with token validation and CORS preflight handling in place
  2. When Google Patents has no PDF link for a granted patent, the extension fetches the PDF through the Worker proxy and produces a citation successfully
  3. A USPTO-fetched patent parses and produces column:line citations identically to a Google Patents-fetched patent
**Plans**: 3 plans

Plans:
- [ ] 06-01-PLAN.md — Cloudflare Worker scaffold (wrangler config, bearer auth, CORS, 3-step ODP orchestration, PDF streaming)
- [ ] 06-02-PLAN.md — Extension-side fallback integration (new message types, three fallback trigger points, offscreen USPTO fetch, manifest update)
- [ ] 06-03-PLAN.md — Deployment + end-to-end verification (deploy Worker, set secrets, update extension constants, human-verify full flow)

### Phase 7: Server-side Cache
**Goal**: Parsed position maps are stored in Cloudflare KV so future users get instant citation results for previously-parsed patents without any PDF download or parse
**Depends on**: Phase 6 (Worker must be deployed and reachable)
**Requirements**: CACH-01, CACH-02, CACH-03, CACH-04
**Success Criteria** (what must be TRUE):
  1. When a user opens a patent that another user has previously parsed, the extension reaches citation-ready state without downloading or parsing a PDF
  2. After a user successfully parses a patent locally, that patent's position map is uploaded to the shared KV cache (fire-and-forget, no UI impact)
  3. If the Cloudflare Worker is unreachable, the extension falls through to the existing PDF parse flow within 3 seconds with no user-visible error
  4. Each patent key is written to KV at most once (existence check before write protects the free-tier write quota)
**Plans**: 3 plans

Plans:
- [ ] 07-01-PLAN.md — Worker KV cache routes (GET /cache read, POST /cache write with existence check, wrangler KV binding)
- [ ] 07-02-PLAN.md — Extension-side cache integration (cache check before PDF fetch, cache hit IndexedDB write, fire-and-forget upload after parse)
- [ ] 07-03-PLAN.md — KV namespace creation, deployment, and end-to-end verification (cache miss->upload->hit lifecycle)
