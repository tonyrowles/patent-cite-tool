## Phase Details

### Phase 8: Test Harness Foundation
**Goal**: A regression-safe test infrastructure exists — pure functions are importable by Vitest, a diverse 30-50 patent fixture corpus is captured, and frozen golden outputs are recorded before any algorithm change begins
**Depends on**: Phase 7 (v1.1 complete — testing the final shipped state)
**Requirements**: TEST-01, TEST-02, TEST-03, TEST-04, TEST-05, TEST-06
**Success Criteria** (what must be TRUE):
  1. Running `npx vitest run` from the project root executes all tests without Chrome API errors
  2. The fixture corpus contains at least 30 test cases covering pre-2000 patents, chemical patents, cross-column selections, repetitive claims, and both short and long selections
  3. Every test case has a frozen expected citation string recorded in a baseline file before any algorithm file is modified
  4. Off-by-one line errors are reported distinctly from total mismatches in test output
  5. Documented accuracy metrics (citation match rate, exact accuracy, confidence calibration) are present before Phase 9 begins
**Plans**: TBD

Plans:
- [x] 08-01: Vitest setup, package.json, export keywords, chrome stubs
- [x] 08-02: Fixture generation script and diverse patent corpus
- [x] 08-03: Golden baseline and accuracy metrics

### Phase 9: Accuracy Audit and Algorithm Fixes
**Goal**: The highest-impact citation failure modes are identified, fixed, and proven not to regress any previously-passing test case
**Depends on**: Phase 8 (frozen golden baseline must exist before any algorithm change)
**Requirements**: ACCY-01, ACCY-02, ACCY-03
**Success Criteria** (what must be TRUE):
  1. A manual accuracy audit across varied patent types identifies the dominant failure patterns with their frequency
  2. Algorithm fixes for the highest-impact failures are implemented and each fix is described in a commit message that references the failure mode it addresses
  3. The full test harness passes after every algorithm change with no previously-passing golden cases broken
  4. Documented accuracy metrics show measurable improvement over the pre-fix baseline
**Plans**: 2 plans

Plans:
- [x] 09-01: Corpus expansion, pre-fix baseline snapshot, per-category accuracy-report script
- [x] 09-02: Gutter contamination fix, wrap-hyphen handling, algorithm improvements with regression validation

### Phase 10: Icon Set and Manifest Updates
**Goal**: The extension has production-quality icons at all required sizes with a toolbar state system that visually communicates parse readiness to the user
**Depends on**: Phase 7 (builds on shipped v1.1 manifest and icon structure)
**Requirements**: ICON-01, ICON-02, ICON-03
**Success Criteria** (what must be TRUE):
  1. The toolbar icon displays at sharp resolution on all platforms including Windows (16, 32, 48, 128px active and inactive variants all present in manifest)
  2. The toolbar icon appears gray on non-patent pages, partial color when a patent page is detected but the position map is not yet parsed, and full color when the position map is parsed and ready
  3. Running `node scripts/generate-icons.mjs` from the project root regenerates all icon PNGs from the source SVG without manual steps
**Plans**: 2 plans

Plans:
- [x] 10-01: Source SVG design, sharp generation script, 12 PNGs (3 states x 4 sizes), manifest 32px entries
- [x] 10-02: Three-state icon system in service worker (gray/partial/full via chrome.action.setIcon with tabId plumbing)

### Phase 11: Options Page Polish
**Goal**: Extension settings are accessible via the right-click options menu with clear save feedback, and the popup is simplified to a launch point
**Depends on**: Phase 10 (icon artwork establishes visual language for options page styling)
**Requirements**: OPTS-01, OPTS-02, OPTS-03, OPTS-04
**Success Criteria** (what must be TRUE):
  1. Right-clicking the extension icon in the Chrome toolbar shows an "Options" item that opens the options page
  2. The options page contains all settings (trigger mode, display mode, patent prefix) and changes are persisted to chrome.storage.sync when saved
  3. After saving settings, the options page displays a "Saved" confirmation that is visible without page reload
  4. The options page displays the current extension version and a clickable link to the privacy policy
**Plans**: 2 plans

Plans:
- [x] 11-01: Options page (options.html + options.js) with settings, auto-save feedback, version/privacy footer, and manifest options_ui entry
- [x] 11-02: Popup settings removal and options page link via chrome.runtime.openOptionsPage()

### Phase 12: Store Listing and Submission
**Goal**: The extension is submitted to the Chrome Web Store with all required assets, a hosted privacy policy, and a fully completed Developer Dashboard privacy section
**Depends on**: Phases 9, 10, 11 (accurate algorithm, production icons, polished options page all required before submission)
**Requirements**: STOR-01, STOR-02, STOR-03, STOR-04, STOR-05
**Success Criteria** (what must be TRUE):
  1. The privacy policy is live at a stable public URL and explicitly addresses patent position map uploads to Cloudflare KV and chrome.storage.sync preference sync
  2. At least one screenshot at exactly 1280x800 showing a real citation displayed on a Google Patents page is ready for upload
  3. A 440x280 promotional tile exists and meets Chrome Web Store dimension requirements
  4. The Developer Dashboard privacy section has all four subsections completed (single purpose description, permission justifications, remote code certification, data use checkboxes)
  5. The extension is submitted and the Developer Dashboard shows the submission as pending review
**Plans**: 2 plans

Plans:
- [x] 12-01: Privacy policy page (GitHub Pages), options.html href update, and extension ZIP packaging
- [ ] 12-02: Store listing text, permission justifications, dashboard guidance, and Chrome Web Store submission

### Phase 13: Offscreen Wrap-Hyphen Fix
**Goal**: The wrap-hyphen normalization fix from Phase 9 is propagated to the duplicate matchAndCiteOffscreen in offscreen.js, ensuring context-menu citations handle wrap-hyphenated HTML selections correctly
**Depends on**: Phase 9 (wrap-hyphen fix must exist to propagate)
**Requirements**: ACCY-02, ACCY-03 (integration gap closure — these requirements are satisfied via text-matcher.js but the offscreen.js code path was missed)
**Gap Closure**: Closes integration gap from v1.2 audit (Phase 9 → offscreen.js wiring) and flow gap (context-menu citation with wrap-hyphenated text)
**Success Criteria** (what must be TRUE):
  1. `matchAndCiteOffscreen` in `offscreen.js` includes the same `normalized.replace(/- ([a-z])/g, '$1')` fix as `matchAndCite` in `text-matcher.js`
  2. Context-menu citations on wrap-hyphenated selections (e.g., "trans- actions") produce correct matches
**Plans**: 1 plan

Plans:
- [ ] 13-01: Propagate wrap-hyphen normalization to matchAndCiteOffscreen in offscreen.js and add offscreen-matcher unit test
