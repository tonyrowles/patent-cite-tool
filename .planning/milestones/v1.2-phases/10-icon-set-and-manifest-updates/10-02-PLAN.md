<objective>
Integrate a three-state toolbar icon system into the service worker that visually communicates parse readiness to the user: gray (non-patent page), partial color (patent detected, fetching/parsing), and full color (parsed and ready to cite).

Purpose: ICON-02 requires the toolbar icon to reflect extension state. Users currently have no visual feedback about whether the extension is ready to generate citations. The three-state system provides at-a-glance status: gray = "not a patent page," partial = "working on it," full = "ready to cite." This relies on the partial and active PNG sets produced by Plan 10-01.

Output: Updated service-worker.js with ICON_PATHS constant, setTabIcon helper function, tabId plumbing through message handlers, and setIcon calls at the four transition points.
</objective>

**Step 1: Add ICON_PATHS constant and setTabIcon helper**

After the existing `PATENT_TYPE` constant block (around line 47), add:

```javascript
// ---------------------------------------------------------------------------
// Icon state paths — used by chrome.action.setIcon for three-state toolbar icon
// Gray/inactive state is handled by manifest default_icon (no setIcon call needed)
// ---------------------------------------------------------------------------

const ICON_PATHS = {
  partial: {
    '16': 'icons/icon-partial-16.png',
    '32': 'icons/icon-partial-32.png',
    '48': 'icons/icon-partial-48.png',
    '128': 'icons/icon-partial-128.png',
  },
  active: {
    '16': 'icons/icon-active-16.png',
    '32': 'icons/icon-active-32.png',
    '48': 'icons/icon-active-48.png',
    '128': 'icons/icon-active-128.png',
  },
};

/**
 * Set the toolbar icon state for a specific tab.
 * Uses tab-scoped setIcon so other tabs are unaffected.
 * Gray state requires no call — Chrome uses manifest default_icon.
 * @param {number|undefined} tabId - The tab to update
 * @param {'partial'|'active'} state - The icon state to set
 */
function setTabIcon(tabId, state) {
  if (!tabId) return;
  chrome.action.setIcon({ path: ICON_PATHS[state], tabId });
}
```

**Step 2: Thread tabId from the message listener to handlers**

The top-level `chrome.runtime.onMessage.addListener` callback receives `sender` as its second argument. `sender.tab?.id` gives the originating tab's ID. Currently, most handlers don't receive this.

Modify the message listener to extract tabId and pass it to the three handlers that need icon state transitions:

```javascript
chrome.runtime.onMessage.addListener((message, sender, sendResponse) => {
  const tabId = sender.tab?.id;

  if (message.type === MSG.PDF_LINK_FOUND) {
    handlePdfLinkFound(message, tabId);
  } else if (message.type === MSG.PDF_LINK_NOT_FOUND) {
    handlePdfUnavailable(message, tabId);
  } else if (message.type === MSG.PDF_FETCH_RESULT) {
    handlePdfFetchResult(message);
  } else if (message.type === MSG.PARSE_RESULT) {
    handleParseResult(message);
  } else if (message.type === MSG.LOOKUP_POSITION) {
    handleLookupPosition(message, sender);
  } else if (message.type === MSG.CITATION_RESULT) {
    handleCitationResult(message);
  } else if (message.type === MSG.USPTO_FETCH_RESULT) {
    handleUsptoFetchResult(message);
  } else if (message.type === MSG.CACHE_HIT_RESULT) {
    handleCacheHitResult(message);
  } else if (message.type === MSG.CACHE_MISS) {
    handleCacheMiss(message);
  } else if (message.type === MSG.GET_STATUS) {
    handleGetStatus(sendResponse);
    return true;
  }
});
```

IMPORTANT: `PDF_LINK_FOUND` and `PDF_LINK_NOT_FOUND` come from the content script (has `sender.tab`). But `PARSE_RESULT`, `CACHE_HIT_RESULT`, and `CACHE_MISS` come from the offscreen document (NO `sender.tab` — offscreen is not a tab). So for these handlers, tabId must be retrieved from storage.

To handle this, store the tabId when `handlePdfLinkFound` or `handlePdfUnavailable` is first called, as part of the `currentPatent` object in `chrome.storage.local`:

In `handlePdfLinkFound`, add `tabId` to the stored object:
```javascript
await chrome.storage.local.set({
  currentPatent: {
    patentId, patentType, kindCode, pdfUrl,
    tabId,  // <-- NEW: store for later setIcon calls
    status: STATUS.FETCHING,
    source: 'google',
    error: null,
  },
});
```

Similarly in `handlePdfUnavailable`, store tabId in the currentPatent object for both the grant and application branches.

Then in `handleParseResult` and `handleCacheHitResult`, read tabId from the stored `currentPatent`:
```javascript
async function handleParseResult(message) {
  const data = await chrome.storage.local.get('currentPatent');
  const patent = data.currentPatent;
  if (!patent) return;
  const tabId = patent.tabId;  // <-- retrieve stored tabId
  // ... existing logic ...
}
```

**Step 3: Add setTabIcon calls at transition points**

Three handlers need setTabIcon calls. Modify each:

1. **handlePdfLinkFound(message, tabId)** — Add `setTabIcon(tabId, 'partial')` right after the `chrome.storage.local.set` call. This transitions gray -> partial when a patent page is detected and fetching begins.

2. **handleParseResult(message)** — In the success branch (where `message.success` is true), add `setTabIcon(tabId, 'active')` after clearing the badge (`chrome.action.setBadgeText({ text: '' })`). This transitions partial -> active when parsing succeeds. Use `patent.tabId` retrieved from storage.

3. **handleCacheHitResult(message)** — Add `setTabIcon(tabId, 'active')` after clearing the badge. This transitions gray -> active on cache hit (instant ready). Use `patent.tabId` retrieved from storage.

**What NOT to change:**
- `handlePdfUnavailable` — Application patents go UNAVAILABLE immediately. Do NOT call setTabIcon here. The amber badge on gray icon is the correct state for "PDF not available." For grant patents taking the cache-check path, the icon will transition via handlePdfLinkFound's normal path if a PDF link was found, or stay gray if not.
- `handlePdfFetchResult` — Icon is already at partial from handlePdfLinkFound. No state change needed during fetch result processing.
- `handleUsptoFetchResult` — Same as above: icon stays at partial during the USPTO fallback attempt. It transitions to active only via handleParseResult on success.
- Error branches in handleParseResult — Icon stays at partial + red/amber badge. Do NOT reset to gray on error — the partial icon + badge communicates "we tried but failed" better than gray + badge (which looks like the extension never activated).
