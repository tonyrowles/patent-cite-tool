**Step 2 — Create the workflow directory and file:**
Create `.github/workflows/ci.yml` with the following exact structure:

```yaml
name: CI

on:
  push:
  pull_request:
    branches: [main]

jobs:
  ci:
    runs-on: ubuntu-latest
    timeout-minutes: 10

    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-node@v4
        with:
          node-version: 22
          cache: 'npm'

      - name: Install dependencies
        run: npm ci

      - name: Build (Chrome + Firefox)
        run: npm run build

      - name: Test — src (Vitest unit)
        run: npm run test:src

      - name: Test — chrome-dist (Vitest chrome)
        run: npm run test:chrome

      - name: Test — firefox-dist (Vitest firefox)
        run: npm run test:firefox

      - name: Test — lint (web-ext lint)
        run: npm run test:lint

      - name: Package Chrome extension
        if: success()
        run: cd dist/chrome && zip -r ../../patent-cite-chrome.zip .

      - name: Package Firefox extension
        if: success()
        run: cd dist/firefox && zip -r ../../patent-cite-firefox.zip .

      - name: Upload Chrome artifact
        if: success()
        uses: actions/upload-artifact@v4
        with:
          name: patent-cite-chrome
          path: patent-cite-chrome.zip
          retention-days: 30

      - name: Upload Firefox artifact
        if: success()
        uses: actions/upload-artifact@v4
        with:
          name: patent-cite-firefox
          path: patent-cite-firefox.zip
          retention-days: 30
```

**Design decisions (Claude's discretion per CONTEXT.md):**
- `timeout-minutes: 10` — build ~15s + tests ~30s; 10 min is generous headroom without risking a 6-hour default hang (HARD-02)
- `retention-days: 30` — reasonable artifact retention; long enough for review cycles, short enough to avoid storage bloat
- Steps fail fast (default behavior) — first test failure halts the job, no `continue-on-error` used, so no artifacts are produced on failure (success criterion #5)
- `if: success()` on packaging/upload steps — explicit intent documentation even though fail-fast already prevents them from running after failure
- Shell `zip` command — pre-installed on ubuntu-latest, no marketplace action dependency needed
- `cd dist/X && zip -r ../../name.zip .` pattern — ensures manifest.json is at ZIP root (not nested under dist/X/)
- Build runs once as its own step before all 4 test steps — avoids re-running build via `npm test` and gives each test suite individual pass/fail visibility in the Actions UI
