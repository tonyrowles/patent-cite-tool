## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| PR branch (untrusted: any contributor can open one) → workflow runner | The workflow runs on `pull_request` triggers, which by GitHub Actions default DOES grant `secrets.GITHUB_TOKEN` to the workflow. Pitfall 1 surface: a malicious PR could exfiltrate the token IF the workflow logs it or runs arbitrary code from the PR's `package.json`. Mitigation: `permissions: contents: read` only; no PR comments or pushes; the workflow runs `npm ci` (which respects the lockfile but COULD run lifecycle scripts from the PR's package.json — the `--ignore-scripts` flag is not added here but is a Phase 47 hardening candidate). |
| `tests/e2e/lib/pdf-verifier.js` override-aware loader → `VERIFIER_PDFJS_PATH` env var → filesystem | The override path is an absolute filesystem path; if a malicious user can set this env var in a non-CI context, they could redirect the verifier to a malicious pdfjs copy. But: the env var is only set by `v40-pdfjs-frame-shift.yml` to `/tmp/old-pdfjs`; no user input reaches this surface. |
| `verifierDeps.pdfjs-dist` pin → Phase 47 audit drift detection | If a future dep-update PR also bumps `verifierDeps.pdfjs-dist` (e.g., a contributor edits both fields in one PR), the separation defense collapses.
