<objective>
Ship `scripts/verify-single-case.mjs` — a thin CLI shim around v3.0's `verifyCitation` from `tests/e2e/lib/pdf-verifier.js` — plus its Vitest unit tests. The shim is the per-case verification primitive consumed by Plan 41-03's `v40-verifier-gate.yml` workflow inside the 3×-consecutive-run loop (VFY-GATE-01 core invariant).

Purpose: Provide a workflow-friendly, exit-code-based contract over the existing verifyCitation function so the gate workflow can call it N times per case from a bash loop and key downstream decisions (pass/fail) off the exit code. The shim does NOT modify the verifier itself — VFY-02 isolation (verifier purity) is strictly preserved. The shim ALSO does NOT post PR comments or perform any GitHub operations — those are the workflow's job, keeping the shim transport-pure and locally testable.

Output: 1 CLI script (~60 LOC) + 1 unit-test file (~80 LOC).
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| GitHub Actions runner bash -> Node CLI | Workflow invokes `node scripts/verify-single-case.mjs --case "$id"` where `$id` comes from `parse-affected-cases.mjs` stdout, which originated in PR-author-controlled PR body. The shim must NOT execute or eval the case ID. |
| pdfjs-dist -> verifyCitation | Verifier code is PINNED to origin/main during workflow runs (Plan 41-03 ships the pin). The shim consumes the pinned verifier unchanged. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-41-02-T1 | Tampering | CLI consumes attacker-controlled --case argv | mitigate | The case ID is used ONLY as a lookup key into TEST_CASES (string equality) and baseline.json (object property access). No shell interpolation, no eval, no fs.writeFile with user-controlled path components (the --output path IS user-controlled but the shim only writes JSON content the shim itself generates — no echo-back of attacker input as code). |
| T-41-02-T2 | Tampering | Shim modifies pdf-verifier.js (VFY-02 violation) | mitigate | Shim is a CONSUMER of verifyCitation, not an extender. Acceptance criterion checks `git diff` of the verifier returns empty after the shim is created. The shim is also forbidden by the verifier-pin step in Plan 41-03 — any shim-side change to pdf-verifier.js gets overwritten by `git checkout origin/main` before the verifier runs in CI. |
| T-41-02-R1 | Repudiation | Workflow log obscures which run failed | mitigate | Per-run `console.log("run N tier=X status=Y")` line; report JSON `runs[]` array preserves chronological run order with the run number explicit; exit code surfaces in workflow log. |
| T-41-02-D1 | DoS | --runs N with huge N exhausts CI minutes | accept | Workflow caller passes --runs 1 in the for-loop (loop iterates 3 times); --runs is a flexibility hatch for local debugging. Workflow-level timeout-minutes (Plan 41-03) caps total CI time. |
| T-41-02-I1 | Information Disclosure | Report JSON leaks PDF paths or text windows in CI logs | accept | The report JSON is an artifact intended for downstream tooling (Phase 42 reporting); cited_text_window is patent text (already public per the test cases being public-domain patents). No PII, no secrets. |

ASVS Categories addressed: V5 (Input Validation — parseArgv validates all flag types, rejects bad input with exit 2), V7 (Error Handling — exit codes 0/1/2 are meaningful and distinct).
</threat_model>
