### Pattern 3: `--push` flag inverted-default semantics
**What:** Phase 42 has `--no-push` (push by default, opt OUT). Phase 46 needs `--push` (don't push by default in subscription mode, opt IN).

**Where:** `scripts/auto-fix.mjs:768-779` (parseArgs options) and `scripts/auto-fix.mjs:729-741` (Step 17 push branch).

**Locked design (CONTEXT decision):**
- Both flags coexist. `--push` and `--no-push` are not mutually exclusive at the parseArgs level (any combo is parseable); the semantics resolution is in the dispatcher.
- Resolution: `noPush` (existing arg) is checked first; if set, skip push regardless of transport. If `--push` is set, push regardless of transport. If neither is set and `transport === 'subscription'`, skip push (the inversion). If neither is set and `transport === 'sdk'`, push (the Phase 42 default — unchanged).
- This preserves byte-identical sdk-transport behavior for all Phase 42 invocations that omit both flags.

**Truth table:**

| transport | --push | --no-push | Push? |
|-----------|--------|-----------|-------|
| sdk | — | — | YES (Phase 42 default — unchanged) |
| sdk | YES | — | YES |
| sdk | — | YES | NO |
| sdk | YES | YES | NO (--no-push wins; document this) |
| subscription | — | — | NO (Phase 46 inversion) |
| subscription | YES | — | YES |
| subscription | — | YES | NO |
| subscription | YES | YES | NO |

**Code shape:**
```js
// Phase 46 — Step 17 push decision
const shouldPush = (() => {
  if (noPush) return false;          // explicit --no-push wins
  if (push) return true;             // explicit --push wins
  return transport === 'sdk';        // default: sdk pushes, subscription does not
})();

if (!shouldPush) {
  process.stdout.write(
    `[auto-fix] branch staged locally; push manually with: ` +
    `git push -u origin ${branchName} ` +
    `(or rerun with --push)\n`,
  );
} else {
  execFileSync('git', ['push', '-u', 'origin', branchName], { stdio: ['ignore', 'inherit', 'inherit'] });
}
```

### Anti-Patterns to Avoid
- **DO NOT remove or weaken the CI guard in `invokeClaudePWithLedger`** (`llm-driver.js:387-393`). Pitfall 8 (#1) of PITFALLS.md identifies this as a load-bearing v3.1 invariant. Phase 46 only ROUTES to this function — it must not modify the guard.
- **DO NOT write the ledger from `build-ledger-dashboard.mjs`**. The dashboard script is read-only — never call `appendLedgerEntry`. Any write would land in the daily snapshot diff and corrupt the audit trail.
- **DO NOT use `new Date()` directly in the dashboard's `Generated:` field**. Use the ledger's `last_invocation_iso` so output is deterministic for unchanged input (prevents noisy "dashboard-only" commits).
- **DO NOT inline dashboard generation into the YAML workflow**. CONTEXT locks the helper-script pattern (`scripts/build-ledger-dashboard.mjs`) for testability.
- **DO NOT auto-redact ledger fields without explicit user confirmation**. CONTEXT recommends NO redaction unless the audit finds actual leakage.

## Common Pitfalls

### Pitfall 1: Removing the subscription CI guard "for v4.0 unification"
**What goes wrong:** A future Phase-46 implementer sees the `if (process.env.CI === 'true' || ...) return ciGate` block at `llm-driver.js:387-393` and assumes "v4.0 needs both transports CI-callable" — deletes the guard.

**Why it happens:** Phase 42 already runs the SDK transport in CI. The subscription guard looks redundant. But Pitfall 8 (#1) of PITFALLS.md is explicit: the guard is what prevents `scripts/e2e-explore.mjs` and the v3.1 triage classifier from accidentally invoking subscription auth on a CI runner (which would burn the developer's personal Max-5 quota on every nightly).

**How to avoid:** Plan 46-01 task ordering MUST place the dispatcher patch BEFORE any unit-test refactor of `llm-driver.js`. The driver itself is NOT touched in Phase 46.

**Warning signs:** Any diff touching `llm-driver.js:387-393` is a code-review blocker.

### Pitfall 2: Dashboard timestamp drift causes noisy daily commits
**What goes wrong:** If `build-ledger-dashboard.mjs` uses `new Date().toISOString()` for the `Generated:` field, then EVERY run of the Phase 40 snapshot workflow produces a non-empty diff for the dashboard markdown — even when the ledger itself didn't change that day. The `git diff --cached --quiet || git commit` guard at `v40-cost-ledger-snapshot.yml:81` would fire daily, committing meaningless "timestamp-only" updates.

**Why it happens:** Default Node.js mental model: "of course you use new Date() for a Generated: line."

**How to avoid:**
1. Derive `Generated:` from the ledger's own `last_invocation_iso` (current-month bucket).
2. Add a Vitest test asserting that `buildDashboardMarkdown(ledger)` is byte-stable across two calls with the same ledger input.
3. Document this requirement in the script's header comment.

**Warning signs:** Daily `[skip ci]` ledger-snapshot commits contain ONLY a `Generated:` line diff in the dashboard file.

### Pitfall 3: `--push` AND `--no-push` set together (operator confusion)
**What goes wrong:** A user passes both flags by mistake (e.g., copy-pastes from two different cookbook examples).

**Why it happens:** parseArgs strict-mode doesn't model mutual exclusion. Both flags are valid bools.

**How to avoid:** Document the precedence (Truth Table in Pattern 3 above). Vitest test for the `--push --no-push` combo asserting it resolves to NO-PUSH.

**Warning signs:** A PR opened from a `subscription` invocation that was supposed to be local-only.

### Pitfall 4: Privacy audit regex false negatives via JSON-escaped boundaries
**What goes wrong:** A regex sweep using `sk-ant-[A-Za-z0-9-]+` on the raw JSON file MAY miss matches embedded inside `error:` fields that have been truncated mid-key, or matches that span a `\n` escape sequence.

**Why it happens:** The ledger's `error` field is a free-form ≤200-char string written by `invokeAnthropicSdkWithLedger` (`llm-driver.js:587`). If an SDK error message happens to include an API key fragment (Anthropic SDK errors generally do NOT include the key, but it's defensive to assume they could), the regex needs to handle JSON-escaped characters.

**How to avoid:**
1. Run the regex against BOTH the raw JSON text AND the value-extracted strings (read the JSON, walk every string leaf, scan each).
2. Use a multiline-aware regex: `/sk-ant-[A-Za-z0-9-]{20,}/g` (the `{20,}` lower-bound rejects accidental matches on test strings like `sk-ant-1`).

**Warning signs:** Audit returns PASS but a manual grep of the file finds something the regex missed.
