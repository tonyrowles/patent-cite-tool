<!-- Contracts the executor needs. Extracted from worker/src/index.js (read directly). -->

The Worker is a single ES-Module file: `export default { async fetch(request, env, ctx) { ... } }`.
Dispatch order in fetch(): (1) OPTIONS preflight, (2) Bearer auth gate, (3) `const url = new URL(request.url); const path = url.pathname;`, (4) `if (path === '/cache') { ... }` block ending at its closing brace, (5) USPTO fallthrough beginning with `const rawPatent = url.searchParams.get('patent') || '';`.

`corsHeaders()` is an existing module-scope helper returning the base CORS headers object; every response spreads `...corsHeaders()`.

`cleanPatentNumber(raw)` is an existing module-scope helper; do NOT call it in /report (it is patent-query-string-specific).

Existing canonical response shapes (copy verbatim from the /cache block):
- Plain-text error: new Response with status code and headers `{ ...corsHeaders(), 'Content-Type': 'text/plain' }`
- JSON success: new Response wrapping JSON.stringify(obj) with headers `{ ...corsHeaders(), 'Content-Type': 'application/json' }`
- X-PCT-Test-Mode guard: `if (request.headers.get('X-PCT-Test-Mode') !== 'true') { /* KV write */ }`

CORS preflight already advertises Allow-Methods `GET, POST, OPTIONS` and Allow-Headers `Authorization, Content-Type, X-PCT-Test-Mode` — POST /report needs no preflight changes.

Worker test harness (worker/tests/test-mode.test.js) imports: `env` from `cloudflare:workers`; `createExecutionContext`, `waitOnExecutionContext` from `cloudflare:test`; `describe, it, expect` from `vitest`; `worker` from `../src/index.js`. Pattern: create ctx, call `await worker.fetch(req, env, ctx)`, then `await waitOnExecutionContext(ctx)` (required for ctx.waitUntil() to settle before assertions). KV list shape: `(await env.BUG_REPORTS.list()).keys[0].name`.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| extension/client -> Worker /report | Untrusted JSON payload crosses here; authenticated only by an extractable Bearer token |
| Worker -> Cloudflare KV | Server-controlled writes; must exclude PII |
| Worker -> Discord webhook | Outbound; user selection text crosses into a chat surface |
| committed repo -> public CRX/XPI + GitHub | Any literal secret here is world-readable |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-1-01 | Information Disclosure | KV report record | mitigate | `buildKvRecord()` is an explicit allowlist (D-08); never copies `ip`/`clientIp`/`userAgent`; `CF-Connecting-IP` flows ONLY to the transient `rl:{ip}` key (60s TTL). Test asserts the record JSON has no `ip` key. |
| T-1-02 | Information Disclosure | Discord webhook URL | mitigate | URL set via `wrangler secret put` only; read via `env.DISCORD_WEBHOOK_URL`; never written to any file. Grep guard: regex `discord\.com/api/webhooks/[0-9]+/` returns 0 results outside `.planning/`. |
| T-1-03 | Denial of Service | /report KV write quota | mitigate | 64KB body cap (D-10, checked via `request.text()` length before parse, not Content-Length per Pitfall 4); IP rate limit 5/60s (LIMIT-02); fingerprint dedup 15-min window (LIMIT-01). |
| T-1-05 | Tampering | Discord embed | mitigate | Selection text truncated to ~200 chars (D-06); Discord payload includes `allowed_mentions: { parse: [] }` to neutralize `@everyone`/`@here` injection. |
| T-1-06 | Spoofing/Tampering | /report request body | mitigate | Bearer `PROXY_TOKEN` gate inherited (D-01, friction not security); required-field validation returns 400 with reason (D-09); category restricted to the 4 frozen `REPORT_CATEGORIES`. |
| T-1-SC | Tampering | npm installs | accept | Zero new packages this plan (sixth-consecutive zero-dep milestone); no install task; no Package Legitimacy Gate needed. |
</threat_model>
