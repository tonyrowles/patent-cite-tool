## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| caller (content script / Phase 4 UI) → buildReportPayload() | Untrusted/loosely-shaped input object (live page snapshot + user-entered category/note) crosses into the pure builder |
| buildReportPayload() output → Worker /report route (Phase 3 transport) | The constructed payload becomes the request body the Worker's allowlist copies into durable KV |

This is a narrow surface: pure functions, no network, no I/O, no chrome, no untrusted deserialization. The only relevant threats are data-integrity / privacy.

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Information Disclosure | buildReportPayload() output | mitigate | PAY-03 forbidden PII fields (ip, clientIp, userAgent) must NEVER appear. Mitigation: the builder constructs output via an explicit ordered object literal containing ONLY the 17 allowlisted keys — it never reads or references the forbidden trio. Pinned by Task 3 SC1 assertion `keys not toContain 'ip'/'clientIp'/'userAgent'` and the Task 2 source-grep acceptance criterion. |
| T-02-02 | Tampering | buildReportPayload() output | mitigate | Unexpected/extra input fields must NOT propagate past the allowlist into the payload (SC1) — an attacker-influenced page snapshot field could otherwise smuggle data into durable KV. Mitigation: explicit ordered object-literal construction (each output key is named individually; spread is used only for the single conditional selectionText key, never for the whole input). Pinned by Task 3 SC1 "extra input field does NOT propagate" assertion. |
| T-02-03 | Tampering | REPORT_CATEGORIES constant | mitigate | A consumer mutating the category allowlist at runtime would let an out-of-allowlist category through D-05. Mitigation: `Object.freeze(REPORT_CATEGORIES)`; pinned by Task 3 `Object.isFrozen` assertion. |
