### 4. Page-Mode Dialog Anatomy

The page-mode dialog is the Phase-4 report dialog rendered in `'page'` mode. The form, radio
group, note textarea, "What's included" panel, button row, and toasts are identical to the
Phase-4 contract. Only the following differ:

#### 4a. Stale-Context Banner (D-01)

Added at the TOP of the panel, before the radio group. Displayed ONLY when `mountContext.mode === 'page'`.

```
Context from your most recent citation: US[patentNumber]
```

If `context.patentNumber` is empty or absent: `"Context from your most recent citation: (none)"`

| Property | Value |
|----------|-------|
| Element | `<div class="cite-report-stale-banner">` |
| Font-size | 11px |
| Font-weight | 400 |
| Color | `#6b7280` (Phase-4 `--text-secondary`) |
| Margin-bottom | 8px (separates banner from the radio group; Phase-4 `sm` token) |
| Background | transparent |
| Set via | `.textContent` â€” NEVER `.innerHTML` (XSS guard; `currentPatent.patentId` is untrusted) |

#### 4b. [Remove Selection Text] Toggle Visibility (D-01)

When `context.selectionText === null` (no live selection â€” always true on the options page):
- The toggle is **hidden** (`style.display = 'none'`), not just inert.
- `includeSelectionText` is locked to `false` in the dialog's local state.
- The "The text you selected" row in the "What's included" panel is NOT shown.

This behavior is determined at dialog build time, before the user interacts.

#### 4c. Panel Positioning (Page Mode)

The panel is in **document flow** â€” no absolute positioning.

```javascript
// Page-mode guard â€” skip the selectionRect positioning block entirely:
if (selectionRect && mountContext.mode === 'shadow') {
  // ... absolute positioning logic (shadow mode only) ...
}
// Page mode: panel is a child of container; no top/left/transform needed
```

The `#reportDialogMount` div's width is constrained by the `.setting-group` horizontal padding
(`24px` each side) inside the `.settings-card` `max-width: 480px` container. The panel's own
`min-width: 220px` / `max-width: 320px` applies within that space.

#### 4d. Focus Trap Root (Page Mode)

Page mode uses `document.activeElement` instead of `shadowRoot.activeElement`.

```javascript
// Page-mode installFocusTrapPage(panelEl, onEscape):
// - getFocusable() queries panelEl for all focusable selectors
// - boundary check: panelEl.contains(document.activeElement)
// - event listener: document.addEventListener('keydown', handleKeydown)
// - cleanup: document.removeEventListener('keydown', handleKeydown)
```

#### 4e. Click-Outside Handler (Page Mode)

```javascript
// Page mode: check container.contains(e.target) instead of citationHost.contains(e.target)
document.addEventListener('mousedown', clickOutsideHandler, { capture: true });
function clickOutsideHandler(e) {
  if (!container.contains(e.target)) dismissDialog();
}
```

The 100ms delay before installing the click-outside handler (present in shadow mode) is
preserved in page mode to prevent the scroll-to-section navigation event from immediately
dismissing the dialog.

#### 4f. Dismiss (Page Mode)

```javascript
function dismissDialog() {
  removeTrap();
  document.removeEventListener('mousedown', clickOutsideHandler, { capture: true });
  panel.remove();
  styleEl.remove();
  // No triggerEl.focus() â€” no trigger element in page mode (D-02 no category pre-selected)
}
```

#### 4g. No Category Pre-Selected (D-02)

All four radio buttons start unchecked. Submit button starts enabled (Phase-4 D-07: category
is optional). The "What's included" panel shows "Problem category: (none selected)" until
the user picks one.

**Rationale for inline confirmation instead of toast (page mode):** The Phase-4 toast functions
(`showSuccessToast` / `showFailureToast`) require a `rect` (bounding rect of the selection / popup
area from the citation UI). In page mode, there is no such rect. The inline confirmation avoids
importing the toast-positioning logic into the options page context and is more appropriate for a
full-tab settings page.

## Anti-Patterns (Phase 5 specific)

The following patterns are explicitly forbidden. They are documented here so the executor can
recognize and avoid them:

| Anti-pattern | Why forbidden | What to do instead |
|-------------|---------------|-------------------|
| `innerHTML` on stale-context banner | XSS â€” `currentPatent.patentId` is untrusted data from extension storage | Use `.textContent` exclusively |
| Amber nudge on green-debug Report button | D-05 LOCKED â€” amber is reserved for failure/yellow/error | Use plain icon `âš‘` with transparent background and `#6b7280` color |
| `shadowRoot.activeElement` in page-mode focus trap | Returns `null` outside a shadow root | Use `document.activeElement` in `installFocusTrapPage` |
| `getCitationHost()` from page-mode mount path | Existing Vitest guard bans this; options page has no citation host | Use the `container` parameter (`#reportDialogMount`) |
| `extractPatentInfo()` from page-mode submit handler | This function reads the Google Patents DOM â€” unavailable on options page | Inject pre-built `context` object from `options.js` (built from `currentPatent` in storage) |
| `selectionRect`-based absolute positioning in page mode | `selectionRect` is `null`; would crash on `null.bottom` | Guard: `if (selectionRect && mountContext.mode === 'shadow')` |
| Showing the [Remove selection text] toggle when `selectionText === null` | No selection text to remove; toggle is meaningless | Hide the toggle entirely (`style.display = 'none'`) and lock `includeSelectionText = false` |
| Duplicating form/payload/submit logic for page mode | D-08 LOCKED â€” drift risk | Thread `mountContext` through `showReportDialog`; shared logic stays single-source |
| `showSuccessToast` / `showFailureToast` in page mode | These functions require a `rect` from the citation UI | Use inline `<p>` confirmation inside `#reportDialogMount` |
| Using `chrome.tabs.create` instead of `openOptionsPage()` | Inconsistent with existing `#settingsLink` pattern | Use `openOptionsPage()` + `pendingOptionsHash` signaling |
