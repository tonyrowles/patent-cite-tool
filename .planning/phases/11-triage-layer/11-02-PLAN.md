<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| KV record fields → gh CLI args | Untrusted user-supplied `patentNumber` flows into `gh issue list/pr list --search` queries (shell command construction). `note`/`selectionText` flow into the Issue body. |
| KV record fields → Issue markdown | A malicious `note` could contain a forged `<!-- kv-key -->` HTML comment attempting to poison dedup. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-11-01 | Tampering | gh search query construction (findExistingIssueByKvKey, isPostFixSuppressed) | mitigate | All search query interpolation escapes single-quote shell context via `replaceAll("'", "'\\''")` (lifted pattern). `patentNumber` originates from a Worker-validated KV record; the escape is defense-in-depth. Enforced by acceptance-criteria grep for the escape pattern. |
| T-11-02 | Tampering | createIssueWithLabels body construction | mitigate | Issue body passed to gh exclusively via `--body-file -` stdin (`{ input: body }`), NEVER concatenated into the shell command string — arbitrary `note`/`selectionText` content cannot break out of the body. Title/label escaped with `replaceAll('"','\\"')`. Enforced by acceptance-criteria grep for `--body-file -`. |
| T-11-03 | Tampering (dedup poisoning) | findExistingIssueByKvKey marker match | mitigate | The dedup marker `<!-- kv-key: report:{fp}:{ts} -->` is constructed by THIS code from the record's canonical `fingerprint`+`timestamp` (Plan 03 builds it), not from user free-text. A forged marker inside a user `note` could at worst match an unrelated kv-key search; mitigated because the search is scoped by the exact kv-key string AND the literal-marker `.includes()` check requires the full constructed marker. The Issue body itself (where the authoritative marker lives) is generated by us, not the user. Residual risk accepted: a user cannot forge a marker for a kv-key they do not own (fp+ts are server-computed). |
| T-11-SC | Tampering | npm/pip/cargo installs | accept | Zero new dependencies (RESEARCH.md). No install tasks; no legitimacy checkpoint required. |
</threat_model>
