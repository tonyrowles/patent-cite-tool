Edit `.github/workflows/ci.yml` to insert two YAML blocks between the `on:` trigger block and the `jobs:` block. Do NOT modify any existing lines — only insert new content.

Insert the following after the `on:` block and before `jobs:`:

```yaml
permissions:
  contents: read

concurrency:
  group: ${{ github.workflow }}-${{ github.head_ref && github.ref || github.run_id }}
  cancel-in-progress: true
```

**Placement rules:**
- `permissions:` block goes first (after `on:`, blank line separator)
- `concurrency:` block goes second (after `permissions:`, blank line separator)
- `jobs:` follows after another blank line separator
- All existing job/step content remains exactly as-is

**Verification after edit:**
1. Confirm `permissions:` appears exactly once at workflow top level (not inside `jobs:`)
2. Confirm `contents: read` is the only permission declared — no `write` scopes anywhere
3. Confirm `concurrency:` appears exactly once at workflow top level
4. Confirm group key contains `github.head_ref && github.ref || github.run_id`
5. Confirm `cancel-in-progress: true` is set
6. Run `npm test` to confirm no regressions (YAML change should not affect local tests)

**Do NOT:**
- Use `cancel-in-progress: ${{ github.ref != 'refs/heads/main' }}` — this does NOT protect pending main runs (see RESEARCH.md Pitfall 1)
- Add `permissions` at job level — workflow level is correct for a single-job workflow
- Add any write permissions — this workflow only reads (checkout) and uploads artifacts (separate API)
