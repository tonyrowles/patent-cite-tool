<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Git tag push → GitHub Actions runner → GitHub Releases API | A `v*` tag push triggers an automated workflow that has `permissions: contents: write` and uses `GITHUB_TOKEN` to publish a Release. The release-publish action is the only effect visible to end users (and the AMO/CWS submission process consumes those ZIPs). A silent regression in this gate (e.g., a removed asset, a muted step, a weakened tag filter) lets a broken or empty release ship. |
| ci.yml `workflow_dispatch` → existing GitHub Release | The manual-fallback path uploads assets to a tag-existing release via `gh release upload --clobber`. The operator-supplied `inputs.tag` is the trust boundary; the `--clobber` flag overwrites without confirmation. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-25-01 | Tampering (integrity) | `.github/workflows/release.yml` trigger block | mitigate | Guard test R2 (Task 2) reads release.yml and asserts the exact `on: push: tags: ['v*']` regex shape. A PR that broadens the filter (e.g., to `'*'` matching arbitrary tags), narrows it (`'v[0-9]*'` excluding pre-releases differently), or removes it entirely fails R2 in the unit-test suite. |
| T-25-02 | Tampering (integrity) | `.github/workflows/release.yml` asset list | mitigate | Guard test R4 (Task 2) asserts both `patent-cite-chrome.zip` and `patent-cite-firefox.zip` appear at least twice in release.yml (packaging + release-create). A PR that drops one of the dists (e.g., removes Firefox packaging) fails R4. |
| T-25-03 | Tampering (integrity) | `.github/workflows/release.yml` gh-release invocation | mitigate | Guard test R5 (Task 2) pins both `gh release create` and `--generate-notes`. A PR that switches to `gh release upload` without first creating the release, or drops `--generate-notes` for a hand-rolled body (which could fail silently if templating is wrong), fails R5. |
| T-25-04 | Repudiation / silent-bypass | release.yml step muting via `continue-on-error: true` | mitigate | Guard test R6 (Task 2) asserts the entire release.yml file contains no `continue-on-error: true` literal. A future PR that mutes the test step (or the package step) to ship a known-broken release fails R6 immediately. Tradeoff: this is a coarse check — if a legitimate `continue-on-error` is ever required for an unrelated step, R6 must be tightened to YAML-parse and check only the critical steps (test, package, release create). |
| T-25-05 | Tampering / cross-workflow interference | `.github/workflows/ci.yml` `on:` block | mitigate | Guard test R7 (Task 2) parses ci.yml's `on:` block and asserts it has no `tags:` sub-key. A PR that adds `tags:` (e.g., `tags: ['v*']` to ci.yml so it ALSO runs on tag pushes) would cause both workflows to fire on a single tag push — duplicate builds, possible race on asset upload, and a violation of SC#4. R7 fails immediately in that case. |
| T-25-06 | Elevation of privilege | `permissions: contents: write` on the release job | accept | The job needs `contents: write` to publish a release via `gh release create`. This is the minimum required scope; narrower scopes (e.g., `metadata: read`) would block the release-create call entirely. Tradeoff: any step inside the job runs with this scope, so a compromised action could theoretically push commits. Mitigation: only first-party actions are used (`actions/checkout@v4`, `actions/setup-node@v4`, plus `gh` CLI which is pre-installed on `ubuntu-latest`). Not pinning these to SHAs is an accepted tradeoff for a small extension project. |
| T-25-07 | Denial of Service (release-process flakiness) | `gh release create` network call | accept | A transient GitHub API failure would fail the release step. The workflow has no automatic retry. Tradeoff: rerunning the failed workflow run from the GitHub UI is trivial; adding retry logic increases workflow complexity without significant value at this scale. Low severity. |
