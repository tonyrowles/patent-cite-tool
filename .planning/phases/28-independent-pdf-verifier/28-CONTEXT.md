<domain>
## Phase Boundary

A second, deliberately-independent code path that re-parses every cited PDF and confirms the selected text actually appears near the cited column:line. This phase adds an **oracle** — an authoritative source of truth that catches citation bugs the golden baseline cannot, because the baseline could have been generated by the same buggy code path it's now being used to validate.

The verifier is the natural diagnostic path for Phase 27's 10 TIMEOUT_PILL deferrals (cases where the extension's offscreen PDF lookup returned "text not found" but the test infrastructure proved the selection was correctly performed in HTML). The verifier will re-parse each of those patents' PDFs in Node and report whether the cited column:line region contains text matching the user's selection.

In scope:
- `tests/e2e/lib/pdf-verifier.js` — independent verifier module
- `tests/e2e/lib/report.js` — structured JSON report writer with the fixed RPT-02 error taxonomy
- `tests/e2e/lib/pdf-snippet.js` — renders PDF page region as cropped PNG on verifier disagreement
- Integration: regression.spec.js calls verifier after each citation observation
- ESLint `no-restricted-imports` rule preventing `tests/e2e/lib/pdf-verifier.js` from importing anything under `src/`
- Per-run `report.json` aggregating all 76 cases with verdicts + error classes + artifact links
- Calibration: verifier's tiered matcher (exact → whitespace-normalized → ±2 line fuzzy) must hit >95% Tier-A/B/C pass rate against the 76-case corpus

Out of scope (deferred):
- CI nightly cron + auto-issue filing — Phase 29
- Worker fault-injection — Phase 30
- LLM exploratory mode — Phase 31
- Poppler/pdftotext second-engine cross-check — deferred to v3.1+ per research recommendation
- Visual regression diffing — v3.1+

<decisions>
## Implementation Decisions

### Verifier Architecture
- **Language: JavaScript ESM (.js)** — matches project convention. Research summary's `.ts` references were illustrative only.
- **PDF parsing engine: `pdfjs-dist` only** (v5.5.207 already installed). Single engine keeps complexity bounded. Poppler/pdftotext cross-check deferred to v3.1 per research; sufficient redundancy comes from the verifier being a separate code path that doesn't import from `src/`.
- **PDF source: re-fetched** from the same upstream the extension uses (Google Patents PDF link, USPTO fallback). The verifier downloads fresh and parses in Node — this is the "independent code path" claim's foundation. Cache locally to `tests/e2e/.pdf-cache/{patentId}.pdf` (gitignored).
- **Tolerance: ±2 line fuzzy** initially. Calibrate against the 76-case corpus during execution. If Tier-A/B/C pass rate < 95%, tune ±N or relax the normalize chain.

### Verifier Independence (VFY-02)
- Verifier code lives in `tests/e2e/lib/pdf-verifier.js`. It MUST NOT import from `src/` (no `src/shared/matching.js`, no constants, no helpers).
- ESLint `no-restricted-imports` rule blocks any `import .../src/...` from `tests/e2e/lib/pdf-verifier.js`. `npm run lint` fails on violation.
- The verifier reimplements its own column:line extraction from pdfjs textContent. Two reasonable paths:
  - **Recommended:** pdfjs `page.getTextContent()` → group items by Y-coordinate → identify column boundaries via x-bimodal analysis → assign each item to (column, line). Then locate the selected text and check it falls within ±N lines of the cited line.
  - Alternative: use pdfjs `getOperatorList()` for raw drawing operations. More complex; recommended path is sufficient.

### Tiered Matcher (VFY-03)
- Tier A — **exact substring** match in the cited line's text.
- Tier B — **whitespace-normalized** match (collapse `\s+` → ` `).
- Tier C — **±N line fuzzy** (default N=2): selected text appears anywhere in `[cited_line - N, cited_line + N]`.
- Tier D — **fail** (text not found within tolerance window).
- Verdict status: `pass` for A/B/C; `disagree` for D.
- Each verdict carries: `tier_used: 'A'|'B'|'C'|'D'`, `cited_text_window: string`, `match_offset_lines: number`.

### Report Schema (RPT-01, RPT-02)
- File: `tests/e2e/artifacts/{run_id}/report.json`
- Top-level shape:
  ```json
  {
    "run_id": "2026-05-15T...",
    "started": "ISO timestamp",
    "ended": "ISO timestamp",
    "summary": {
      "total": 76,
      "passed": 65,
      "skipped": 11,
      "failed": 0,
      "by_error_class": { "WRONG_CITATION": 0, ... }
    },
    "cases": [
      {
        "id": "US11427642-spec-short-1",
        "status": "passed" | "failed" | "skipped",
        "errorClass": null | "WRONG_CITATION" | "EXTENSION_NOT_LOADED" | ...,
        "citation": "1:26-27",
        "verifier_verdict": {
          "status": "pass" | "disagree",
          "tier_used": "A",
          "match_offset_lines": 0,
          "cited_text_window": "..."
        },
        "artifacts": {
          "screenshot": "{caseId}-screenshot.png" | null,
          "dom": "{caseId}-dom.html" | null,
          "pdf_snippet": "{caseId}-pdf-snippet.png" | null
        }
      },
      ...
    ]
  }
  ```
- Error class taxonomy (RPT-02 — fixed strings):
  - `EXTENSION_NOT_LOADED` — extension failed to attach
  - `NO_CITATION_PRODUCED` — extension ran but produced no citation
  - `WRONG_CITATION` — citation differs from baseline AND verifier disagrees
  - `UI_BROKEN` — pill never attached / shadow DOM not accessible
  - `VERIFIER_DISAGREE` — citation matches baseline but verifier independent check failed
  - `GOOGLE_DOM_DRIFT` — pre-flight DOM probe failed
  - `USPTO_API_DRIFT` — Worker/USPTO fallback returned unexpected shape
  - `FLAKE` — transient failure (retry succeeds)

### PDF Snippet Rendering (DIAG-03)
- Module: `tests/e2e/lib/pdf-snippet.js`
- Dependencies (new dev deps): `canvas` (node-canvas) for pdfjs rendering target, `sharp` (already installed) for crop + PNG write.
- Triggered only on `VERIFIER_DISAGREE`. The harness:
  1. Fetches the PDF for the patent
  2. Renders the page containing the cited column:line at ~150 DPI
  3. Crops the region around the cited line (configurable margin, default 100px above + below)
  4. Writes PNG to `tests/e2e/artifacts/{run_id}/{case_id}-pdf-snippet.png`
- Renders only when a human needs to adjudicate the disagreement — keeps artifact directory small.

### Verifier Invocation Pattern
- The regression spec calls `verifyCitation({patentId, selectedText, observedCitation})` inside the per-test catch/after path, after `getCitation`.
- Verifier verdict + citation observation are both written to the per-case report entry.
- On verifier disagreement, the catch block additionally calls `renderPdfSnippet(...)` to produce DIAG-03 artifact.
- The verifier is async (PDF parse takes ~2-5s per patent); cache parsed text within a single run so repeated cases on the same patent reuse parsed output.

### Phase 27 Deferrals as Phase 28's First Test
The 10 TIMEOUT_PILL cases from Phase 27 (`27-09-SUMMARY.md`) are the immediate use case:
- US11427642-claims-1, US11427642-repetitive
- US4723129-claims, US4723129-claims-repetitive
- US5371234-chemical-cross-col, US5371234-claims
- US5440748-claims, US5440748-repetitive
- US7346586-claims-repetitive
- US8352400-claims

For each, the verifier should:
1. Re-parse the PDF
2. Check whether the test-cases.js `selectedText` appears in the expected column:line region
3. If YES (verifier confirms): the extension has a bug — log `VERIFIER_DISAGREE` (the extension said no, the verifier says yes)
4. If NO (verifier also can't find it): the test fixture has bad data — log `WRONG_CITATION` against the *test-cases* row, recommend updating selectedText
5. Either way: produce a PDF snippet showing what the verifier saw

After Phase 28 ships, re-enable the 10 deferred cases in regression.spec.js (move from `TIMEOUT_PILL_DEFERRED_IDS` to fully-asserted set OR document a different remediation per case).

<specifics>
## Specific Ideas

- The verifier's PDF fetch should respect the same URL pattern the extension uses: try Google Patents PDF link first (`https://patents.google.com/patent/{id}.pdf` or the linked PDF URL extracted from the page); fall back to USPTO eGrant API via the Cloudflare Worker. Cache by patent ID.
- The verifier should log every tier it tried in the verdict so debugging is straightforward.
- Add ESLint rule to `.eslintrc` or `eslint.config.js`:
  ```
  no-restricted-imports: [error, { patterns: [{ group: ['../../../src/**', '../../src/**'], message: 'Verifier must not import from src/ — independence claim (VFY-02)' }] }]
  ```
  Scoped only to `tests/e2e/lib/pdf-verifier.js`.
- `report.json` is written incrementally (per-case append + final overwrite) so a partial run still produces a usable report.
