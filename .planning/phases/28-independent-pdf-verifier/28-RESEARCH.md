# Phase 28: Independent PDF Verifier — Research

**Researched:** 2026-05-14
**Domain:** Node-side PDF parsing + tiered text matching as an oracle for the extension's offscreen PDF pipeline
**Confidence:** HIGH

## Summary

Phase 28 ships a deliberately-independent oracle that re-parses every cited PDF in Node and confirms the selected text appears near the cited column:line. The verifier is the diagnostic path for Phase 27's 10 deferred TIMEOUT_PILL cases and the long-term guard against citation regressions the golden baseline cannot catch (because the baseline was generated by the same code path it's now used to validate).

The recommended implementation: a single ESM module `tests/e2e/lib/pdf-verifier.js` that imports `pdfjs-dist/legacy/build/pdf.mjs` directly (not via any shared helper), implements its own column/line inference from raw `getTextContent()` items (mirroring the production algorithm conceptually but in fresh code), applies a 4-tier matcher (Tier A exact → Tier B whitespace-normalized → Tier C ±2-line fuzzy → Tier D fail), and returns a structured verdict. Independence is enforced by an ESLint flat config with a `no-restricted-imports` rule scoped to that one file. On Tier-D verdicts, a separate `tests/e2e/lib/pdf-snippet.js` renders the cited page region via `pdfjs-dist`'s built-in canvas factory (which auto-uses `@napi-rs/canvas` when available — a 1-package install, no native cairo/pango required) and crops via `sharp`.

The dominant planning risks are: (1) **ESLint is not currently installed** — Phase 28 must add it and initialize a flat config from scratch, not "add a rule to an existing config"; (2) **`@napi-rs/canvas` is the recommended Node canvas backend** for v3.0 — it ships prebuilt binaries (no apt-get cairo/pango required), and pdfjs v5.5.207 already declares it as an optional dependency that its built-in `canvasFactory` auto-detects; (3) **DOMMatrix/Path2D polyfills must use the `legacy` build** (`pdfjs-dist/legacy/build/pdf.mjs`) — the non-legacy build assumes browser globals; (4) **calibration is empirical** — the ±2-line tolerance is a starting hypothesis; the verifier must be run against the 65 currently-passing cases first and tuned to hit >95% Tier A/B/C before being trusted as an oracle.

**Primary recommendation:** Implement the verifier with `pdfjs-dist/legacy/build/pdf.mjs` + `@napi-rs/canvas` (optional dep, lazy-loaded) + `sharp`. Use a Wave-0 calibration task that runs the verifier against the 65 passing cases before it's wired into `regression.spec.js`. Add ESLint flat config + `no-restricted-imports` rule scoped to `tests/e2e/lib/pdf-verifier.js`.

- Verifier code lives in `tests/e2e/lib/pdf-verifier.js`. It MUST NOT import from `src/` (no `src/shared/matching.js`, no constants, no helpers).

### Pattern 1: Verifier as Pure Function

**What:** `verifyCitation({patentId, selectedText, observedCitation})` is a single async function that takes inputs, returns a verdict object. No side effects in the verifier itself except (a) PDF fetch and (b) reading/writing the PDF cache. The verifier does NOT write `report.json` — the *caller* does.

**Why:** Same module is reusable by Phase 29's CI workflow and Phase 31's LLM exploratory mode without re-engineering.

**Verdict shape:**
```js
{
  status: 'pass' | 'disagree',
  tier_used: 'A' | 'B' | 'C' | 'D',
  cited_text_window: string,          // The text the verifier saw at the cited line
  match_offset_lines: number,         // 0 if Tier A/B, ±N if Tier C, null if Tier D
  reason: string,                     // human-readable: "matched at line 27 (Tier A)" or "selected text not found within ±2 lines of cited 12:18"
  pdf_path: string,                   // absolute path to the cached PDF used
  pages_parsed: number,               // how many PDF pages were read (for performance debug)
  duration_ms: number,
}
```

### Pattern 2: PDF Fetch → Cache → Parse Pipeline

**What:**
```
verifyCitation
  ├── ensureCachedPdf(patentId)        // pdf-fetch.js
  │     ├── path = `tests/e2e/.pdf-cache/${patentId}.pdf`
  │     ├── if fs.existsSync(path) → return path
  │     ├── url = `https://patents.google.com/patent/${patentId}.pdf`
  │     ├── fetch(url) → write to path → return path
  ├── parsedRef = parsedCache.get(patentId)         // in-memory Map for the run
  ├── if !parsedRef → parsedRef = await parsePdf(path) ; parsedCache.set(...)
  ├── runMatcher(parsedRef, selectedText, observedCitation)
  └── return verdict
```

**When to use:** Always. The in-memory cache is critical because multiple test cases share patents (e.g., 6 cases per `US11427642`); parsing once per patent saves ~2-5s × 5 repeats per patent.

### Pattern 3: Tiered Matcher with Verdict Trail

**What:** Try Tier A, then B, then C. Return the first success with the tier name. On Tier D (fail), return verdict with `status: 'disagree'`.

```js
function runMatcher(parsed, selectedText, citation) {
  const citedRange = parseCitation(citation);   // "12:18-19" → {col: 12, lineStart: 18, lineEnd: 19}

  // Get the cited window — exact lines from the parsed PDF
  const exactLines = parsed.linesFor(citedRange.col, citedRange.lineStart, citedRange.lineEnd);
  const exactText = exactLines.map(l => l.text).join(' ');

  // Tier A — exact substring
  if (exactText.includes(selectedText)) {
    return { status: 'pass', tier_used: 'A', cited_text_window: exactText, match_offset_lines: 0, ... };
  }

  // Tier B — whitespace-normalized
  const ws = s => s.replace(/\s+/g, ' ').trim();
  if (ws(exactText).includes(ws(selectedText))) {
    return { status: 'pass', tier_used: 'B', ..., match_offset_lines: 0 };
  }

  // Tier C — ±N-line fuzzy
  const N = 2;
  const fuzzyLines = parsed.linesFor(citedRange.col, citedRange.lineStart - N, citedRange.lineEnd + N);
  const fuzzyText = fuzzyLines.map(l => l.text).join(' ');
  if (ws(fuzzyText).includes(ws(selectedText))) {
    // Compute offset for diagnostics
    const offset = findMatchLineOffset(fuzzyLines, selectedText, citedRange.lineStart);
    return { status: 'pass', tier_used: 'C', ..., match_offset_lines: offset };
  }

  // Tier D — fail
  return { status: 'disagree', tier_used: 'D', cited_text_window: exactText, match_offset_lines: null, ... };
}
```

**Why deliberately simpler than `src/shared/matching.js`:** The production matcher has 5 tiers including Levenshtein fuzzy and gutter-tolerant fallback (Tier 5). The verifier must NOT reproduce that complexity — the goal is to catch bugs in the production matcher, not validate that "both matchers produce the same answer." Tier C with `±2 line` window + whitespace-normalize is the floor of what a human would expect to count as "the cited text is there."

### Pattern 4: Independent Column/Line Inference (mirroring `src/offscreen/`)

**What:** The verifier reimplements column/line assignment from raw `getTextContent()` items. It MIRRORS the production logic conceptually but writes fresh code in `pdf-verifier.js`. Direct comparison with `src/offscreen/position-map-builder.js`:

| Step | Production (`src/offscreen/`) | Verifier (`pdf-verifier.js`) |
|------|-------------------------------|------------------------------|
| 1. Per-page items | `pdf-parser.js extractTextFromPdf` | Inline call to `page.getTextContent()` then filter empty items |
| 2. Two-column detection | `isTwoColumnPage` (bimodal x-coord) | Reimplement: same idea, simpler threshold — `leftCount / rightCount ratio > 0.3` |
| 3. Column boundary | `findColumnBoundary` (gap-width scan) | Reimplement: simpler — find the largest 0-count x-bucket in the middle half of page |
| 4. Header/footer filter | `filterHeadersFooters` (top 90pt, bottom 40pt) | Reimplement: same thresholds, fresh code |
| 5. Printed column numbers | `extractPrintedColumnNumbers` (header digit scan) | Reimplement: read top-90pt strip, find lone 1-3 digit numbers on each side of page mid, require `right === left + 1` and `left % 2 === 1` |
| 6. Gutter line-number filtering | `filterGutterLineNumbers` (mult-of-5 in range 5-65 near boundary) | Reimplement: same heuristic, fresh code |
| 7. Line clustering | `clusterIntoLines` (y ± 3pt tolerance) | Reimplement: same 3pt tolerance |
| 8. Line number assignment | `assignLineNumbersByGrid` OR `assignLineNumbers` | Reimplement BOTH: use gutter-grid when ≥2 markers found; fallback to median-spacing blank-line counting |

**Anti-pattern to avoid:** Copy-paste from `src/offscreen/position-map-builder.js`. ESLint's `no-restricted-imports` blocks `import`, but copy-paste defeats the independence goal. Reviewers should diff-test: if `pdf-verifier.js` contains a function whose body is byte-identical to `position-map-builder.js`, that's a finding. The algorithms are the same, the implementations must be fresh.

**Why bother reimplementing the same algorithm?** The CONTEXT.md is explicit (VFY-02): "Verifier reimplements its own column:line extraction." This is the architectural decision the milestone hinges on — a bug in `position-map-builder.js`'s column-detection that the production matcher inherits would also pass through any verifier that imported that code. Fresh implementation forces the verifier through a different bug surface.

### Pattern 5: Page Identification from Cited (Column, Line)

**What:** Citations are document-wide column numbers (1, 2, 3, ..., 60+) and line numbers (1-65 per column). The verifier must identify which PDF page contains the cited (column, line). Two strategies:

1. **Use the verifier's own PositionMap** — after building entries with `{page, column, lineNumber, text, ...}`, search for entries matching the cited column+line and read off `entry.page`. This is the production approach and is reliable.
2. **Read printed column numbers from page headers** — but this only works if `extractPrintedColumnNumbers` succeeded for that page. If column numbers are graphics-rendered (e.g., `US10203551`), it fails.

**Recommend:** Strategy 1. Build the full PositionMap, then `posMap.findEntry({col, line})`.

### Pattern 6: Incremental Report Writer

**What:** `report.js` exposes an `appendCase(caseEntry)` that:
1. Reads existing `report.json` (creates a skeleton if missing)
2. Replaces or appends the entry matching `caseEntry.id`
3. Updates `summary.passed/failed/skipped/by_error_class` counters
4. Writes back the full JSON

**Why read-modify-write per case:** A partial run still produces a usable report. If the suite times out at case 50/76, you have 50 entries. The atomicity unit is the case, not the run.

**Performance:** With 76 cases and a ~10KB report, read-modify-write is negligible (<5ms per case). Not worth a streaming JSON writer.

**Concurrency:** Playwright runs `workers: 1` (set in `playwright.config.js` per Phase 26 — extensions require persistent context, no parallelism). So no file locking needed.

### Anti-Patterns to Avoid

- **Importing from `src/`** — explicitly blocked by ESLint, but also: don't import `tests/fixtures/*.json` (the production PositionMaps). Use only the live-fetched PDF.
- **Hand-rolled fetch retry** — keep it simple. One `fetch()`, on failure log + return `disagree` with a `PDF_FETCH_FAIL` reason. CI flake retry is Phase 29's job.
- **Using `getOperatorList()` instead of `getTextContent()`** — works, but exposes much more surface area for engine quirks. `getTextContent()` is the well-trodden path. CONTEXT.md lists `getOperatorList` as Claude's discretion; recommend NOT using it.
- **Reading the cited text from `src/offscreen/`'s cached fixtures** — that's the production code's interpretation of the PDF. The verifier must build its own.
- **Writing `report.json` from inside `pdf-verifier.js`** — Single Responsibility. Verifier returns a verdict; the spec calls `report.appendCase(...)`.
- **Re-rendering the same page multiple times for snippet** — cache the page-render result by patent ID + page number across the run.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| PDF parsing | Custom PDF byte parser | `pdfjs-dist/legacy/build/pdf.mjs` | A real-world PDF parser is years of work. pdfjs handles fonts, encodings, ligatures, CIDs, encryption, etc. |
| Node canvas backend | `canvas` (node-canvas) with native build steps | `@napi-rs/canvas` (prebuilt N-API binaries) | Ships precompiled `.node` files per OS/arch. No `apt-get install libcairo2-dev`. pdfjs v5 declares it as `optionalDependencies`. [CITED: `node_modules/pdfjs-dist/package.json`] |
| PNG cropping | Pixel arithmetic | `sharp` `.extract().png().toBuffer()` | sharp is already in deps; battle-tested; supports streaming. |
| HTTP fetch | `node-fetch`, `axios`, `got` | `globalThis.fetch` | Node 22+ ships stable fetch. No deps needed for one URL fetch. |
| ESM lint enforcement of an import boundary | grep in CI | ESLint flat config + `no-restricted-imports` rule scoped to `tests/e2e/lib/pdf-verifier.js` | CONTEXT.md is explicit; ESLint's `files` glob + per-file rule overrides are the canonical pattern. [CITED: https://eslint.org/docs/latest/rules/no-restricted-imports] |
| Levenshtein-fuzzy substring | Reimplement `src/shared/matching.js`'s `fuzzySubstringMatch` | The 4-tier matcher with ±N-line window | The verifier is intentionally simpler; reproducing the production fuzzy matcher defeats the independence claim. |
| PDF text extraction in two engines | Add Poppler/`pdftotext` as a second engine | Single engine (pdfjs); rely on verifier-is-different-code-path-not-different-engine independence | Locked by CONTEXT.md. Poppler deferred to v3.1. |
| JSON-report mutation safety | Lock file or transactional store | Synchronous read-modify-write (`workers: 1` in Playwright config) | Phase 26 already locked single-worker. No concurrency, no locking needed. |

**Key insight:** Phase 28 is a *thin* phase — most of the heavy lifting (PDF parsing, image manipulation, HTTP fetch) is delegated to well-supported libraries. The verifier's *novel* code is column/line inference + the 4-tier matcher. Keep these two pieces small, well-named, and unit-testable; everything else is glue.

## Common Pitfalls

### Pitfall 1: Using the wrong pdfjs build path

**What goes wrong:** `import { getDocument } from 'pdfjs-dist'` (default entry) throws `ReferenceError: DOMMatrix is not defined` in Node. The default entry expects a browser global polyfilled by the browser; Node has neither DOMMatrix nor Path2D nor ImageData. [CITED: [mozilla/pdf.js#19847 — DOMMatrix is not defined](https://github.com/mozilla/pdf.js/discussions/19847)]

**Why it happens:** pdfjs-dist ships two build flavors — the default for modern browsers (smaller, assumes browser polyfills) and a `legacy` build that polyfills what it can and tolerates missing Node-incompatible APIs.

**How to avoid:** Always `import from 'pdfjs-dist/legacy/build/pdf.mjs'` in the verifier. Same import path is used by the Mozilla pdf2png example. [CITED: https://github.com/mozilla/pdf.js/blob/master/examples/node/pdf2png/pdf2png.mjs]

**Warning signs:** First-run `npm test` failure with `ReferenceError: DOMMatrix is not defined` — immediately switch import path.

### Pitfall 2: Calibration assumption (±2 lines is too tight OR too loose)

**What goes wrong:** Verifier reports many false `VERIFIER_DISAGREE` verdicts on cases the production extension correctly cited — because pdfjs in Node extracts text with slightly different whitespace/ligature handling than the extension's offscreen pipeline, AND the verifier's tier B (whitespace-normalize) doesn't fully bridge the gap. OR: verifier reports `pass` on cases where the cited text is actually 10 lines away — because Tier C window is wider than needed and the same word recurs.

**Why it happens:** The 65 currently-passing cases in `regression.spec.js` are the ground truth for "the production extension's column:line citation is correct." The verifier's matcher must agree with the production extension on >95% of those cases to be trusted as an oracle on the 10 deferred cases.

**How to avoid:** Add a **Wave-0 calibration task** to the Phase 28 plan. The task runs the verifier against the 65 currently-passing cases (NOT the 10 deferred ones) and reports per-case tier verdicts. If Tier A/B/C pass rate < 95%, the plan author tunes one of:
- ±N (try ±3 if many cases miss by 1-2 lines)
- Add Tier B' (alphanumeric-only normalize) — strip punctuation and dashes
- Lower-case in Tier B/C
- Strip wrap-hyphens before comparison (the production matcher does this — see `src/shared/matching.js` lines 656-658)

Only after the verifier hits >95% on known-good cases is it wired into `regression.spec.js` for adjudication of the deferred cases.

**Warning signs:** Verifier disagree rate >5% on the 65 passing cases. That's not a 10-deferred-case problem; that's a verifier calibration problem.

### Pitfall 3: Verifier inherits the same engine bugs the extension has

**What goes wrong:** Both the extension and the verifier use pdfjs to extract text. If pdfjs has a bug for a specific PDF (e.g., out-of-order text items, ligature mis-mapping), both will see the same garbage text. The verifier reports `pass` against the garbage; the production extension produces a wrong citation. Garbage-in-garbage-out from both — the verifier theatre is exposed.

**Why it happens:** Same engine. CONTEXT.md acknowledges this: "Single engine keeps complexity bounded. Poppler/pdftotext cross-check deferred to v3.1."

**How to avoid:** Three independence layers compensate:
1. **Different invocation surface** — Node vs offscreen-document-in-extension. Different worker setup, different font-handling.
2. **Different code path** — no shared functions; fresh column/line inference; ESLint-enforced.
3. **Different matching algorithm** — production has 5 tiers including Levenshtein; verifier has 4 tiers, all substring-based.

**What this does NOT solve:** A true pdfjs bug in `getTextContent` for a specific PDF. The mitigation for that is the v3.1+ Poppler cross-check, explicitly deferred.

**Warning signs:** Verifier passes 100% of the 65 cases AND passes 10/10 of the deferred cases. That's suspicious — the production extension said the deferred cases couldn't find the text; the verifier should disagree on at least some. If it doesn't, calibrate harder or accept that the engine is the limit.

### Pitfall 4: Page identification ambiguity for cited (column, line)

**What goes wrong:** Citation `12:18-19` doesn't directly say which PDF page contains column 12 lines 18-19. The verifier must build the document-wide column→page mapping. If the verifier's column-detection disagrees with the production's column-detection, the verifier looks at the wrong page entirely — and reports `VERIFIER_DISAGREE` for the wrong reason.

**Why it happens:** `extractPrintedColumnNumbers` can return `null` for pages where pdfjs doesn't extract the printed column number cleanly. The production code has a fallback (infer from page order). The verifier needs the same fallback OR an even simpler one.

**How to avoid:** Mirror `position-map-builder.js`'s primary + fallback pass pattern (see `buildPositionMap` lines 730-779). Primary: iterate two-column pages, read printed column numbers, validate sequential (`left === expectedLeftCol`). Fallback: if primary yields zero entries, infer columns from page order starting at column 1.

**Warning signs:** Verifier's parsed-PDF map has fewer columns than the production map. Mid-run sanity check: `assert posMap.maxColumn >= productionPosMap.maxColumn - 2`. Don't fail-loud, but log it.

### Pitfall 5: PDF fetch from Google Patents (auth + rate-limit)

**What goes wrong:** `fetch('https://patents.google.com/patent/US11427642.pdf')` returns HTML (a redirect page or a 404 page) instead of a PDF, because that URL pattern isn't the actual PDF host.

**Why it happens:** The extension finds the PDF link by scraping the page DOM for `a[href*="patentimages.storage.googleapis.com"]` (verified in `src/content/content-script.js:54`). The actual PDF lives on Google's storage CDN, and the URL is per-patent (with image-process IDs in the path). There is no clean URL convention.

**How to avoid:** Two options for the planner to pick:
- **Option A (recommended):** Load the Google Patents page via plain `fetch` (HTML body), regex-extract the first `https://patentimages.storage.googleapis.com/...\.pdf` href, then fetch that URL. Cache by patent ID. No extension load required.
- **Option B:** Use the same URL the extension's content script captured during the live spec run. Pass it through to the verifier via the per-case context. Tighter coupling but no second HTTP request.

**Option A is preferred** for verifier independence (it doesn't depend on a successful extension run). The "fetch HTML → regex for PDF link" is identical to what `findPdfLink` does in `src/content/content-script.js`, so the verifier's fetch logic is conceptually the same algorithm in fresh code.

**Warning signs:** First `fetch().arrayBuffer()` is 5KB instead of 500KB+ → it's an HTML page, not a PDF. Check `Content-Type: application/pdf` before writing to cache.

### Pitfall 6: `@napi-rs/canvas` install reports success but pdfjs can't find it

**What goes wrong:** `npm install -D @napi-rs/canvas` succeeds but `page.render()` throws `Cannot find module '@napi-rs/canvas'`. [CITED: [mozilla/pdf.js#19145](https://github.com/mozilla/pdf.js/issues/19145)]

**Why it happens:** pdfjs's built-in `canvasFactory` lazy-loads `@napi-rs/canvas`. If the install is incomplete (npm cache corruption, missing prebuilt binary for the platform), the lazy-load throws at render time, not at install time.

**How to avoid:** Add an install probe step in the Phase 28 plan:
```bash
node -e "import('@napi-rs/canvas').then(m => console.log('OK:', !!m.createCanvas))"
```
Run after `npm install`. If it fails, the plan must surface the failure (don't proceed to building the snippet renderer).

**Warning signs:** Test run reaches the snippet path and throws `Cannot find module '@napi-rs/canvas'`. Install probe would have caught it earlier.

### Pitfall 7: PDF cache stale after Google Patents republishes a patent

**What goes wrong:** Google Patents occasionally re-issues PDFs (corrections, page-rotation fixes). Cached PDF is the old one; verifier reports `pass` on stale text; live extension reports based on the new PDF; verdicts diverge.

**Why it happens:** No cache invalidation strategy is documented.

**How to avoid:** Two options for the planner:
- **Option A:** Cache forever; document that a maintainer clears `tests/e2e/.pdf-cache/` after a baseline recalibration. Cheapest, fine for v3.0.
- **Option B:** Cache with `If-Modified-Since` using the file mtime. Adds complexity.

Recommend Option A. It's already implicit in CONTEXT.md ("Cache by patent ID"). Document the manual-clear in `tests/e2e/README.md` (or a comment in `pdf-fetch.js`).

### Pitfall 8: ESLint flat config without an installed ESLint

**What goes wrong:** Phase 28 writes `eslint.config.js` but `npm run lint` fails with `eslint: command not found`. Or worse: the rule never enforces because there's no script invoking eslint.

**Why it happens:** Project does NOT have ESLint installed (`ls node_modules/eslint` returns "not found"). The only existing lint script is `npm run test:lint` which runs `npx web-ext lint` (Firefox manifest validator) — entirely unrelated.

**How to avoid:** Phase 28 plan must include both:
1. `npm install -D eslint@<pinned>` — first-time install
2. New npm script: `"lint": "eslint tests/e2e/lib/"` (or scope wider — Claude's discretion)
3. Document in plan that this is a NEW config, not an extension of an existing one
4. The plan's verification step must include `npm run lint` returning exit 0

**Warning signs:** Phase 28 PR is reviewed and someone asks "where's the no-restricted-imports rule enforced?" and there's no answer because no `lint` script exists.

## Code Examples

### Verified pattern: Render PDF page to PNG in Node

```js
// Source: Mozilla pdf2png example (https://github.com/mozilla/pdf.js/blob/master/examples/node/pdf2png/pdf2png.mjs)
import { getDocument } from 'pdfjs-dist/legacy/build/pdf.mjs';
import path from 'node:path';

const CMAP_URL = path.resolve('node_modules/pdfjs-dist/cmaps/');
const STANDARD_FONT_DATA_URL = path.resolve('node_modules/pdfjs-dist/standard_fonts/');

async function renderPageToPng(pdfBuffer, pageNum) {
  const pdfDocument = await getDocument({
    data: pdfBuffer,
    cMapUrl: CMAP_URL + '/',
    cMapPacked: true,
    standardFontDataUrl: STANDARD_FONT_DATA_URL + '/',
  }).promise;

  const page = await pdfDocument.getPage(pageNum);
  const canvasFactory = pdfDocument.canvasFactory;  // pdfjs auto-uses @napi-rs/canvas
  const viewport = page.getViewport({ scale: 150 / 72 });   // 150 DPI
  const canvasAndContext = canvasFactory.create(viewport.width, viewport.height);

  const renderTask = page.render({
    canvasContext: canvasAndContext.context,
    viewport,
  });
  await renderTask.promise;

  const pngBuffer = canvasAndContext.canvas.toBuffer('image/png');
  await pdfDocument.destroy();
  return pngBuffer;
}
```

### Verified pattern: Text content with positions

```js
// Source: src/offscreen/pdf-parser.js lines 49-92 (production code — verifier mirrors but doesn't import)
import { getDocument } from 'pdfjs-dist/legacy/build/pdf.mjs';

async function extractTextWithPositions(pdfBuffer) {
  const pdf = await getDocument({ data: pdfBuffer }).promise;
  const pages = [];

  for (let i = 1; i <= pdf.numPages; i++) {
    const page = await pdf.getPage(i);
    const textContent = await page.getTextContent();
    const viewport = page.getViewport({ scale: 1.0 });

    const items = textContent.items
      .filter((it) => it.str && it.str.trim().length > 0)
      .map((it) => ({
        text: it.str,
        x: it.transform[4],          // x-coordinate
        y: it.transform[5],          // y-coordinate (PDF origin = bottom-left)
        width: it.width,
        height: it.height,
        fontName: it.fontName,
        hasEOL: it.hasEOL,
      }));

    pages.push({
      pageNum: i,
      items,
      pageWidth: viewport.width,
      pageHeight: viewport.height,
    });
  }

  await pdf.destroy();
  return pages;
}
```

The verifier's version of this function lives in `pdf-verifier.js` and does NOT import from `src/offscreen/pdf-parser.js`. The algorithm is the same; the code is fresh.

### Verified pattern: Incremental report writer

```js
// Source: tests/e2e/lib/report.js (NEW for Phase 28)
import fs from 'node:fs';
import path from 'node:path';

const ERROR_CLASSES = [
  'EXTENSION_NOT_LOADED', 'NO_CITATION_PRODUCED', 'WRONG_CITATION',
  'UI_BROKEN', 'VERIFIER_DISAGREE', 'GOOGLE_DOM_DRIFT',
  'USPTO_API_DRIFT', 'FLAKE',
];

function emptySummary() {
  return {
    total: 0, passed: 0, skipped: 0, failed: 0,
    by_error_class: Object.fromEntries(ERROR_CLASSES.map(k => [k, 0])),
  };
}

export function appendCase(reportPath, caseEntry) {
  let report;
  try {
    report = JSON.parse(fs.readFileSync(reportPath, 'utf8'));
  } catch {
    report = {
      run_id: path.basename(path.dirname(reportPath)),
      started: new Date().toISOString(),
      ended: null,
      summary: emptySummary(),
      cases: [],
    };
  }

  const existingIdx = report.cases.findIndex(c => c.id === caseEntry.id);
  if (existingIdx >= 0) report.cases[existingIdx] = caseEntry;
  else report.cases.push(caseEntry);

  // Recompute summary from scratch — cheaper than incremental delta
  report.summary = emptySummary();
  for (const c of report.cases) {
    report.summary.total++;
    if (c.status === 'passed') report.summary.passed++;
    else if (c.status === 'skipped') report.summary.skipped++;
    else report.summary.failed++;
    if (c.errorClass && ERROR_CLASSES.includes(c.errorClass)) {
      report.summary.by_error_class[c.errorClass]++;
    }
  }
  report.ended = new Date().toISOString();

  fs.mkdirSync(path.dirname(reportPath), { recursive: true });
  fs.writeFileSync(reportPath, JSON.stringify(report, null, 2));
}
```

## Open Questions

   - Decision deferred to plan; doesn't block research.

   - Separate script is cleaner — calibration is a one-shot exercise per verifier-algorithm change, not a per-run gate.
   - Recommendation: separate script. Plan decision.

5. **Should the verifier handle USPTO eGrant API fallback like the extension does, or only Google Patents?**
   - CONTEXT.md is explicit: "verifier hits Google Patents directly" (open Q #7 in the additional context). Worker is unchanged in Phase 28.
   - Resolution: Google Patents only for v3.0. Document the gap.

## Security Domain

`security_enforcement` flag is absent from config.json (default = enabled). The phase is test-side only with no user data handling, so most ASVS categories don't apply. Threats are limited to:

### Applicable ASVS Categories

| ASVS Category | Applies | Standard Control |
|---------------|---------|------------------|
| V2 Authentication | No | No auth surface — fetching public PDFs |
| V3 Session Management | No | No sessions |
| V4 Access Control | No | Read-only public data |
| V5 Input Validation | Yes (light) | Validate citation strings parse cleanly (`^\d+:\d+(-\d+(:\d+)?)?$`); cap PDF buffer size (reject >50MB) to prevent OOM on malformed downloads |
| V6 Cryptography | No | Never hand-roll. No crypto in scope. |
| V12 File Handling | Yes | PDF cache writes must use `patentId.pdf` as filename (already restricted to `^[A-Z]{2}\d+[A-Z]?\d*$` by `patentIdFromCaseId` regex in `regression.spec.js`); no user-controlled paths |

### Known Threat Patterns for {Node verifier + Google Patents fetch}

| Pattern | STRIDE | Standard Mitigation |
|---------|--------|---------------------|
| Untrusted PDF triggers pdfjs RCE | Tampering | pdfjs is sandboxed (no shell exec, no `fs` access except via explicit ops). Mitigated by upstream; verifier doesn't add a new surface. |
| Malicious filename from patent ID causes path traversal in cache | Tampering | Patent ID regex `^[A-Z]{2}\d+[A-Z]?\d*$` rejects `../` and unsafe chars. Already enforced in `regression.spec.js`. Re-enforce in `pdf-fetch.js`. |
| Huge PDF (gigabyte) exhausts disk/memory | DoS | Reject `Content-Length > 50MB` before download; `fetch` Response streaming + size cap. |
| Test-only secrets in `report.json` | Info disclosure | `report.json` contains: case IDs, citations, snippets of patent text, file paths. NO credentials. Patent text is already public. No info-disclosure concern. |
| Cache pollution between test runs | Tampering | Verifier reads from cache by exact patent ID; no per-run isolation needed because PDFs are deterministic-per-patent. |

**Conclusion:** Low security surface. Input-validation on the citation string and the patent-ID-as-filename is sufficient. No new ASVS V6 (crypto) considerations.
