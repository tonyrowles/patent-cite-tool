    import { createExecutionContext, waitOnExecutionContext } from 'cloudflare:test';
    import { describe, it, expect } from 'vitest';
    import worker from '../src/index.js';

    const TEST_TOKEN = 'test-token'; // must match miniflare.bindings.PROXY_TOKEN

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Extension/offscreen → Cloudflare Worker | HTTPS request crosses untrusted Internet; bearer-token authed |
| Worker handler → KV binding | Internal Cloudflare data-plane; new write-suppression flag added here |
| Vitest test → Worker module (in-process via Miniflare) | Isolated per-file in-memory KV; no production data flow |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-30-1 | Tampering | worker/src/index.js POST /cache guard | accept | `X-PCT-Test-Mode` is NOT a secret. Any caller with the bearer token can already DoS the cache (existing risk pre-this phase). The header lets such a caller suppress their OWN write only — no new privilege, no new data access. The production extension code never sets the header (verified by `grep -r "X-PCT-Test-Mode" src/` returning empty), so production behavior is unchanged. ASVS L1 V5 input-validation not relevant — header value is a constant string equality check, no parsing or injection surface. |
| T-30-2 | Information Disclosure | PROXY_TOKEN in worker/tests/test-mode.test.js | accept | The literal `'test-token'` in the test file is a Miniflare-only mock value injected via `miniflare.bindings` — NOT the real production token. The real `PROXY_TOKEN` is a Cloudflare Worker secret (set via `wrangler secret put`); Miniflare cannot read production secrets. No information disclosure. |
| T-30-3 | Tampering | env.PATENT_CACHE (Miniflare KV) in tests | accept | Miniflare KV is a per-test-file in-memory namespace, fully isolated from the production KV binding (id `6e7af6faa9c340fdb8120036913b00b5`). Test writes cannot reach production. Verified by Cloudflare docs §"vitest-integration/test-apis". |
| T-30-4 | Tampering | CORS preflight `Access-Control-Allow-Headers` | accept | Adding `X-PCT-Test-Mode` to the allow-list does NOT change auth semantics — the header is still subject to the existing Bearer-token check above it (line 144-153). A malicious cross-origin caller without the bearer token still gets 401. CORS allow-list grants no privilege; it only permits the browser to deliver the header. |
</threat_model>
