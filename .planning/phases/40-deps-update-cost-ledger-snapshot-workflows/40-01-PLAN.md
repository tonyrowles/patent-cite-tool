<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| GitHub Actions runner → repo `main` branch | The runner has `contents: write` and pushes a commit to main. Any malicious modification of the workflow file (or of the `tests/e2e/lib/llm-ledger.js` module the runner imports) becomes a vector for arbitrary commits to main. |
| Scheduled cron trigger surface | A typo (`0 */2 * * *` instead of `0 2 * * *`) would run the workflow 12× per day instead of 1×; multiplied by `git push` to main, this is a $4,320/month-equivalent blast-radius noise burst (Pitfall 2 / Phase 38). |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-40-01-01 | Tampering | `.github/workflows/v40-cost-ledger-snapshot.yml` (cron schedule) | mitigate | YAML-level static-grep test S1 pins exact literal `cron: '0 2 * * *'`. Any drift fails the test before merge. |
| T-40-01-02 | DoS | github-actions[bot] push loop | mitigate | `[skip ci]` token in commit message (S7 pins it). Without `[skip ci]`, every snapshot commit would re-trigger `ci.yml` (which runs on push to default branch), recursively. |
| T-40-01-03 | Tampering | Empty commit creating audit-trail noise | accept | Per CONTEXT discretion line 51: "always-commit is simpler and the [skip ci] prevents downstream impact". The `git diff --cached --quiet || git commit` idempotent guard prevents EMPTY commits (no-op when ledger unchanged) — the test S6 pins the guard. |
| T-40-01-04 | Information Disclosure | Snapshot commit message leaks spend data publicly | accept | Spend totals are non-sensitive aggregate dollars-per-day numbers; the committed ledger file already has the same data more granularly. Phase 47 CLEANUP audit re-evaluates the privacy posture. |
| T-40-01-05 | Elevation of Privilege | Use of PAT instead of GITHUB_TOKEN | mitigate | Workflow uses `secrets.GITHUB_TOKEN` only (set via `env: GH_TOKEN`); test S5 indirectly asserts this by pinning the workflow doesn't reference any `secrets.*PAT*` literal — covered by test S5 negative space. |
| T-40-01-06 | Tampering | Bot identity spoofing | mitigate | `git config user.name "github-actions[bot]"` + `41898282+github-actions[bot]@users.noreply.github.com` pinned by test S9. GitHub's UI recognizes this exact email as the bot identity; any other email leaves commits attributed to a real user. |
